# Stock Reports Lab MVP 이슈 단위 분해

상태: 초안
작성 기준: `docs/proposal/stock-report-mvp-product-scope.md`

## 1. 문서 목적

이 문서는 제품 기능 범위를 구현 이슈로 전환하기 쉽게 작업 단위로 재구성한다.

각 항목은 GitHub Issue 또는 구현 계획 문서로 분리할 수 있는 단위다. 세부 API URL, DB 컬럼, 클래스 설계는 각 이슈의 하위 구현 계획에서 확정한다.

## 2. MVP 구현 이슈

1. 데이터 모델과 마이그레이션 기반을 정의한다.
   - 종목, 일봉 가격, 시장 지수, 기술지표, 신호, 리포트 리비전, 종목 분석, 업종 분석의 저장 기준을 마련한다.
   - Python 배치가 사용할 테이블은 Spring Flyway가 생성한다.

2. Python 배치 실행 기반을 구성한다.
   - 매 거래일 19:00 Asia/Seoul 기준 실행되는 배치 구조를 마련한다.
   - KRX 거래일이 아닌 경우 실패가 아니라 정상 종료로 처리한다.
   - 중복 실행을 방지한다.

3. KRX 종목 목록과 업종 정보를 수집한다.
   - FinanceDataReader의 `StockListing("KRX")`를 사용해 종목 목록과 당일 거래량을 조회한다.
   - FinanceDataReader의 `StockListing("KRX-DESC")`를 사용해 종목 설명과 업종을 조회한다.

4. 분석 대상 종목 200개를 선정한다.
   - 종가 1,000원 이상 종목만 포함한다.
   - 거래량 내림차순 상위 200개 종목을 선정한다.
   - 분석 범위가 KRX 전체가 아님을 이후 화면에서 표시할 수 있게 한다.

5. 종목별 일봉 가격 데이터를 수집하고 갱신한다.
   - FinanceDataReader의 KRX 수정주가 OHLCV를 사용한다.
   - 신규 분석 종목은 최근 2년 일봉을 확보한다.
   - 기존 분석 종목은 최근 10일을 재조회해 갱신한다.
   - 동일 종목과 거래일의 가격 데이터가 중복 생성되지 않게 한다.

6. KOSPI와 KOSDAQ 시장 지수 데이터를 수집한다.
   - KOSPI는 `KS11`, KOSDAQ은 `KQ11`을 사용한다.
   - 시장 개요와 지수 차트에서 사용할 일봉 데이터를 준비한다.

7. MACD와 Stoch MACD를 계산한다.
   - MACD는 fast 12, slow 26, signal 9 기준으로 계산한다.
   - Stoch MACD는 K 14, D 3, smooth K 3 기준으로 계산한다.
   - 최초 계산 버전은 `stoch-macd-v1`로 관리한다.

8. 골든크로스 신호를 판정한다.
   - 최근 세 거래 일봉 안에서 K선이 D선을 아래에서 위로 돌파한 사건을 찾는다.
   - 동일 신호가 중복 저장되지 않게 한다.
   - 신호는 매수 추천이나 상승 보장으로 취급하지 않는다.

9. 종목별 분석 상태를 저장한다.
   - 신호 발견, 신호 없음, 분석 데이터 부족, 데이터 준비 중, 데이터 갱신 실패, 당일 거래 없음, 분석 실패를 구분한다.
   - 한 종목의 실패가 전체 리포트 공개를 막지 않게 한다.

10. 리포트 리비전 게시 정책을 구현한다.
    - 첫 수집 회차 종료 후 정상 종목 결과를 최초 리비전으로 공개한다.
    - 실패 종목 재시도 종료 후 변경분을 최종 리비전으로 한 번만 공개한다.
    - 사용자에게는 거래일별 최종 활성 리비전만 노출한다.

11. 업종 분석 데이터를 생성한다.
    - 분석 종목 200개 기준으로 업종 통계를 생성한다.
    - 업종별 종목 수, 최근 골든크로스 종목 수, 골든크로스 발생 비율을 계산한다.
    - 분석 완료 종목만 발생 비율의 분모로 사용한다.

12. 시장 전체 AI 요약을 생성한다.
    - 거래일별 시장 전체 요약 1건만 생성한다.
    - 입력은 계산된 시장 지수 통계와 분석 종목 통계로 제한한다.
    - AI 실패 또는 지연 중에도 금융 데이터 리포트는 제공한다.

13. 최신 장 마감 리포트 조회 API를 제공한다.
    - 기본 조회는 최신 활성 리비전을 반환한다.
    - 리포트 기준 거래일과 분석 범위를 함께 제공한다.
    - 오늘 리포트가 아직 공개되지 않은 경우 사용자에게 표시할 상태를 구분할 수 있게 한다.

14. 과거 장 마감 리포트 조회 API를 제공한다.
    - 서비스 운영 시작일 이후 거래일별 리포트를 조회한다.
    - 휴장일, 미생성일, 서비스 시작 전 날짜의 처리를 구분할 수 있게 한다.

15. 골든크로스 보고서 API를 제공한다.
    - 활성 리비전 기준 골든크로스 종목 목록을 반환한다.
    - 신호 발생일, 현재가, K값, D값, 업종을 제공한다.

16. 전체 분석 종목 API를 제공한다.
    - 활성 리비전 기준 분석 종목 200개를 반환한다.
    - 가격, 거래량, 분석 상태, 최근 신호를 제공한다.
    - 검색과 정렬을 지원한다.

17. 종목 상세 API를 제공한다.
    - 종목 기본정보, 일봉 캔들, 거래량, MACD, Stoch MACD, 신호 마커를 제공한다.
    - 요청 기준일 이후의 데이터가 섞이지 않게 한다.

18. 업종 현황 API를 제공한다.
    - 활성 리비전 기준 업종 트리맵과 업종 통계를 반환한다.
    - 업종 통계가 분석 종목 200개 기준임을 표시할 수 있게 한다.

19. 프론트엔드 공통 레이아웃과 라우팅을 구성한다.
    - 시장 개요, 골든크로스 보고서, 전체 분석 종목, 종목 상세, 업종 현황, 과거 리포트 화면으로 이동할 수 있게 한다.

20. 시장 개요 화면을 구현한다.
    - KOSPI와 KOSDAQ 지수 차트를 표시한다.
    - 분석 종목 기준 시장 통계를 표시한다.
    - AI 시장 요약 상태와 내용을 표시한다.

21. 골든크로스 보고서 화면을 구현한다.
    - 최근 세 거래 일봉 안에 골든크로스가 발생한 종목을 표시한다.
    - 신호 발생일과 보고서 기준 거래일을 구분해 표시한다.

22. 전체 분석 종목 화면을 구현한다.
    - 분석 종목 200개 목록을 표시한다.
    - 검색과 정렬을 제공한다.
    - 종목별 분석 상태를 구분해 표시한다.

23. 종목 상세 화면을 구현한다.
    - 1W, 1M, 3M, 1Y 일봉 캔들 차트를 표시한다.
    - 거래량, MACD, Stoch MACD 시계열을 표시한다.
    - 골든크로스 발생 마커를 표시한다.
    - 과거 리포트 기준으로 진입한 경우 기준 거래일 이후 데이터는 표시하지 않는다.

24. 업종 현황 화면을 구현한다.
    - 분석 종목 200개 기준 업종 트리맵을 표시한다.
    - 업종별 종목 수, 최근 골든크로스 종목 수, 발생 비율을 표시한다.

25. 과거 리포트 조회 화면을 구현한다.
    - 서비스 운영 시작일 이후 거래일별 리포트를 조회할 수 있게 한다.
    - 선택한 리포트의 기준 거래일을 명확히 표시한다.

26. 로컬 관심종목 기능을 구현한다.
    - 관심종목을 브라우저 로컬 저장소에 저장한다.
    - 로그인과 서버 동기화는 제공하지 않는다.

27. 사용자 표시 원칙을 화면 전반에 적용한다.
    - 모든 화면에 데이터 기준 거래일을 표시한다.
    - 분석 범위가 종가 1,000원 이상 및 거래량 상위 200개 기준임을 표시한다.
    - 골든크로스를 투자 추천으로 오해하지 않게 표시한다.

28. 장애 상태와 부분 공개 경험을 구현한다.
    - 데이터 준비 중, 데이터 갱신 실패, 분석 데이터 부족, AI 분석 중, AI 생성 지연을 구분해 표시한다.
    - 일부 종목 실패가 전체 화면을 빈 화면으로 만들지 않게 한다.

29. 핵심 계산과 게시 정책을 검증한다.
    - 고정된 가격 데이터에서 지표와 신호 계산 결과가 반복 실행마다 동일해야 한다.
    - 동일 종목과 거래일의 가격 데이터가 중복되지 않아야 한다.
    - 동일 신호가 중복 저장되지 않아야 한다.
    - 활성 리비전은 거래일별 하나만 사용자에게 노출되어야 한다.

30. MVP 제외 범위가 구현에 섞이지 않았는지 검증한다.
    - 실시간 데이터, 분봉, 로그인, 서버 저장 관심종목, 종목별 AI 설명, 대체 데이터 소스 fallback이 MVP에 포함되지 않았는지 확인한다.
    - Spring API와 프론트엔드가 기술지표를 재계산하지 않는지 확인한다.

## 3. 구현하지 않는 기능

아래 항목은 MVP 구현 이슈로 만들지 않는다. 구현 중 해당 기능이 섞이지 않았는지 검증 기준으로만 사용한다.

1. KRX 전체 종목을 기준으로 한 MVP 통계를 제공하지 않는다.
2. 골든크로스를 매수 추천 또는 상승 보장으로 표현하지 않는다.
3. Spring API 또는 프론트엔드에서 기술지표를 재계산하지 않는다.
4. 사용자에게 이전 리비전 비교 화면을 제공하지 않는다.
5. 휴장일을 실패 리포트로 생성하지 않는다.
6. AI 실패를 이유로 금융 데이터 리포트 공개를 중단하지 않는다.
7. Python 배치가 실행 중 DB 스키마를 생성하거나 변경하지 않는다.
8. MVP에서 인증 기반 개인화 기능을 제공하지 않는다.
9. MVP에서 운영 백업과 다중 서버 구성의 상세 정책을 구현하지 않는다.

## 4. 추후 확장 기능 이슈

아래 항목은 MVP 구현 이슈에는 포함하지 않고, 추후 확장 기능으로 별도 백로그에서 관리한다.

1. 분석 종목 수 확대
2. KRX 전체 종목 분석
3. 추가 기술지표 도입
4. 추가 신호 도입
5. 계산 버전 v2 이상 추가
6. 종목별 AI 설명
7. 로그인 기반 관심종목
8. 기기 간 관심종목 동기화
9. 외국인 및 기관 수급 데이터 연동
10. 대체 데이터 소스 fallback
11. 실시간 현재가와 장중 데이터
12. 실시간 호가와 체결 스트림
13. 분봉 차트
14. 과거 리포트 소급 생성
15. 이전 리비전 비교 및 변경 원인 조회
16. 제한 병렬 수집
17. 운영 모니터링 고도화
18. 외부 백업과 보존 정책
19. 다중 서버 또는 분리 배포
20. API 계약 버전 관리
21. 상세 성능 최적화와 캐시 전략
