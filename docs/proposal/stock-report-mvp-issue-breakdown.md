# Stock Reports Lab MVP 이슈 단위 분해

상태: 초안
작성 기준: `docs/proposal/stock-report-mvp-product-architecture-baseline.md`

## 1. 문서 목적

이 문서는 제품 기능 범위를 구현 이슈로 전환하기 쉽게 작업 단위로 재구성한다.

각 항목은 GitHub Issue 또는 구현 계획 문서로 분리할 수 있는 단위다. 세부 API URL, DB 컬럼, 클래스 설계는 각 이슈의 하위 구현 계획에서 확정한다.

## 2. MVP 구현 이슈

1. [server] 데이터 모델과 마이그레이션 기반을 정의한다.
   - 종목, 일봉 가격, 시장 지수, 기술지표, 신호, 리포트 리비전, 종목 분석, 업종 분석의 저장 기준을 마련한다.
   - 기술지표 저장 기준에는 MACD, Stoch MACD, 이동평균을 포함한다.
   - 기술지표는 계산 버전별로 저장하고 계산법 변경 시 기존 값을 덮어쓰지 않게 한다.
   - 리포트 리비전은 해당 리비전이 참조하는 계산 버전을 저장한다.
   - 과거 리포트 맥락 보존을 위해 리비전별 종목명, 시장, 업종, 선정 순위, 선정 기준 거래량 스냅샷 저장 기준을 마련한다.
   - 업종 분석은 업종 규모, 면적 기준, 시가총액 합계, 거래대금 합계, 평균 등락률, 신호 수, 비율, 비율 분모 수, 제외 수를 저장할 수 있게 한다.
   - Python 배치가 사용할 테이블은 Spring Flyway가 생성한다.

2. [worker] Python 배치 실행 기반을 구성한다.
   - 매 거래일 19:00 Asia/Seoul 기준 실행되는 배치 구조를 마련한다.
   - KRX 거래일이 아닌 경우 실패가 아니라 정상 종료로 처리한다.
   - 중복 실행을 방지한다.
   - 종목별 수집은 순차 처리한다.
   - 종목별 요청 제한시간은 30초로 둔다.
   - 5개 종목이 연속으로 시간 초과하면 공급자 장애로 보고 현재 회차를 중단한다.
   - 실패·미처리 종목은 10분 간격으로 최대 3회 재시도한다.

3. [worker] KRX 종목 목록과 업종 정보를 수집한다.
   - FinanceDataReader의 `StockListing("KRX")`를 사용해 종목 목록과 당일 거래량을 조회한다.
   - FinanceDataReader의 `StockListing("KRX-DESC")`를 사용해 종목 설명과 업종을 조회한다.
   - 현재 종목 메타데이터 갱신과 리비전별 종목명, 시장, 업종 스냅샷 저장을 분리한다.

4. [worker] 분석 대상 종목 200개를 선정한다.
   - 종가 1,000원 이상 종목만 포함한다.
   - 거래량 내림차순 상위 200개 종목을 선정한다.
   - 리포트 재현을 위해 리비전별 선정 순위와 선정 기준 거래량을 저장할 수 있게 한다.
   - 분석 범위가 KRX 전체가 아님을 이후 화면에서 표시할 수 있게 한다.

5. [worker] 종목별 일봉 가격 데이터를 수집하고 갱신한다.
   - FinanceDataReader의 KRX 수정주가 OHLCV를 사용한다.
   - 신규 분석 종목은 최근 3년 일봉을 확보한다.
   - 기존 분석 종목은 최근 10일을 재조회해 갱신한다.
   - 분석 대상에서 빠졌다가 재진입한 종목은 보유 이력을 재사용하고 부족한 구간만 보완한다.
   - 동일 종목과 거래일의 가격 데이터가 중복 생성되지 않게 한다.
   - 과거 일봉과 지표는 차트와 신호 계산 맥락을 위해 적재하지만 출시 이전 날짜의 장 마감 리포트는 소급 생성하지 않는다.

   수용 기준:
   - 신규 분석 종목은 최근 3년 일봉을 확보한다.
   - 기존 분석 종목 재실행 시 동일 종목과 거래일의 가격 행이 중복되지 않는다.
   - 초기 적재 데이터만으로 서비스 시작일 이전 장 마감 리포트가 생성되지 않는다.

6. [worker] KOSPI와 KOSDAQ 시장 지수 데이터를 수집한다.
   - KOSPI는 `KS11`, KOSDAQ은 `KQ11`을 사용한다.
   - 시장 개요와 지수 차트에서 사용할 일봉 데이터를 준비한다.

7. [worker] MACD와 Stoch MACD 신호 지표를 계산한다.
   - MACD는 fast 12, slow 26, signal 9 기준으로 계산한다.
   - Stoch MACD는 K 14, D 3, smooth K 3 기준으로 계산한다.
   - Stoch MACD는 MACD 선 자체를 high·low·close 입력으로 사용해 계산한다.
   - RSI와 Stoch RSI는 MVP 보조 지표로 계산하지 않는다.
   - 최초 계산 버전은 `stoch-macd-v1`로 관리한다.
   - 리포트와 신호는 `stoch-macd-v1` 결과를 참조한다.
   - 리포트 리비전에 참조 계산 버전을 기록해 한 리포트 안에서 같은 계산 버전의 지표와 신호만 사용하게 한다.

   수용 기준:
   - 고정된 가격 fixture로 반복 실행해도 MACD와 Stoch MACD 결과가 동일하다.
   - `docs/prototypes/stock-analysis/scripts/stoch_macd_golden_cross_scanner.py`와 동일한 Stoch MACD 계산 의미론을 따른다.
   - 한 리포트 안에서 서로 다른 계산 버전의 지표와 신호가 섞이지 않는다.
   - 계산 결과는 Spring API나 프론트엔드에서 재계산하지 않고 저장된 값을 조회한다.

8. [worker] 이동평균 지표를 계산한다.
   - 이동평균은 거래일 기준 MA5, MA20, MA60, MA120, MA240을 계산한다.
   - 이동평균은 충분한 과거 일봉 이력을 기준으로 Python에서 계산한다.
   - 사용자 화면 라벨은 1주, 1개월, 3개월, 6개월, 1년으로 표시할 수 있게 한다.

   수용 기준:
   - 고정된 가격 fixture로 반복 실행해도 이동평균 결과가 동일하다.
   - MA5, MA20, MA60, MA120, MA240 결과가 저장된다.
   - Spring API와 프론트엔드는 이동평균을 재계산하지 않는다.

9. [worker] 골든크로스 신호를 판정한다.
   - 최근 세 거래 일봉 안에서 K선이 D선을 아래에서 위로 돌파한 사건을 찾는다.
   - 신호 식별자는 종목, 신호 종류, 발생일, 계산 버전 기준으로 관리한다.
   - 동일 신호가 중복 저장되지 않게 한다.
   - 신호는 매수 추천이나 상승 보장으로 취급하지 않는다.

   수용 기준:
   - 골든크로스 신호는 Stoch MACD K선이 D선을 아래에서 위로 돌파한 사건만 저장한다.
   - 달력일이 아니라 최근 세 거래 일봉을 기준으로 판정한다.
   - 동일 종목, 신호 종류, 발생일, 계산 버전의 신호가 중복 저장되지 않는다.

10. [worker] 종목별 분석 상태를 저장한다.
   - 신호 발견, 신호 없음, 분석 데이터 부족, 데이터 준비 중, 데이터 갱신 실패, 당일 거래 없음, 분석 실패를 구분한다.
   - 당일 거래 없음 상태의 종목은 해당 거래일의 신규 골든크로스로 발행하지 않는다.
   - 한 종목의 실패가 전체 리포트 공개를 막지 않게 한다.

11. [worker/server] 리포트 리비전 게시 정책을 구현한다.
    - 첫 수집 회차 종료 후 정상 종목 결과를 최초 리비전으로 공개한다.
    - 실패 종목 재시도 종료 후 변경분을 최종 리비전으로 한 번만 공개한다.
    - 최초 리비전은 공개 가능한 활성 리비전으로 노출하고, 최종 리비전이 생성되면 거래일의 활성 리비전을 최종 리비전으로 교체한다.
    - 사용자 기본 조회에는 항상 거래일별 활성 리비전 하나만 노출한다.
    - 종목 목록 자체를 가져오지 못한 경우 당일 리포트 리비전을 만들지 않고 생성 지연 상태로 처리하며 직전 활성 리포트를 유지한다.

    수용 기준:
    - 첫 수집 회차 종료 후 정상 종목 결과가 최초 리비전으로 공개된다.
    - 재시도 성공 때마다 새 리비전을 만들지 않는다.
    - 모든 데이터 재시도가 끝난 뒤 변경분을 모아 최종 리비전 한 번만 추가한다.
    - 최종 리비전이 생성되면 같은 거래일의 이전 리비전은 보존하되 사용자 기본 조회의 활성 대상에서는 제외된다.
    - 사용자 기본 조회에는 거래일별 활성 리비전 하나만 노출된다.
    - 종목 목록 조회 실패 시 당일 리포트는 생성 지연 상태로 표시되고 직전 활성 리포트가 유지된다.

12. [worker] 업종 트리맵 면적 기준을 리비전 단위로 결정한다.
    - 트리맵 면적은 업종별 시가총액 합계를 우선 사용한다.
    - 시가총액 결측으로 기본 기준을 안정적으로 적용할 수 없으면 같은 리비전 전체에 대해 업종별 거래대금 합계를 사용한다.
    - 시가총액과 거래대금 모두 안정적으로 적용할 수 없으면 같은 리비전 전체에 대해 업종별 분석 종목 수를 사용한다.
    - 실제 사용한 면적 기준은 리비전별 업종 분석 데이터와 API 응답에 `market_cap`, `trade_value`, `stock_count` 중 하나로 저장·노출할 수 있게 한다.

    수용 기준:
    - 리비전 안의 모든 업종은 동일한 트리맵 면적 기준을 사용한다.
    - 면적 기준은 시가총액, 거래대금, 분석 종목 수 순서로 결정된다.
    - 실제 사용한 면적 기준이 업종 분석 데이터와 API 응답에 저장·노출된다.

13. [worker] 업종 분석 데이터를 생성한다.
    - 분석 종목 200개 기준으로 업종 통계를 생성한다.
    - 업종별 시가총액 합계를 산출할 수 있게 한다.
    - 업종별 거래대금 합계를 산출할 수 있게 한다.
    - 업종 종목의 일간 등락률 단순 평균을 계산한다.
    - 업종별 종목 수, 최근 골든크로스 종목 수, 골든크로스 발생 비율을 계산한다.
    - 신호 판정 완료 종목만 발생 비율의 분모로 사용한다.
    - 분석 데이터 부족, 데이터 갱신 실패, 당일 거래 없음으로 비율 분모에서 제외된 종목 수를 집계한다.

    수용 기준:
    - 업종 통계는 해당 거래일 분석 종목 200개만 대상으로 한다.
    - 업종별 규모, 면적 기준, 시가총액 합계, 거래대금 합계, 평균 등락률, 최근 골든크로스 종목 수, 비율, 비율 분모 수, 제외 수가 저장된다.
    - 분석 데이터 부족, 데이터 갱신 실패, 당일 거래 없음 종목은 비율 분모에서 제외되고 제외 수에 포함된다.

14. [worker] 시장 전체 AI 요약을 비동기로 생성한다.
    - 거래일별 시장 전체 요약 1건만 생성한다.
    - 입력은 계산된 시장 지수 통계와 분석 종목 통계로 제한한다.
    - 종목 데이터 재시도가 모두 끝난 뒤 AI 요약을 생성한다.
    - AI 실패 시 1분, 5분, 15분 간격으로 최대 3회 재시도한다.
    - AI 실패 또는 지연 중에도 금융 데이터 리포트는 제공한다.
    - AI 상태와 내용 변경만으로 금융 데이터 리비전을 추가하지 않는다.

15. [server] 최신 장 마감 리포트 조회 API를 제공한다.
    - 기본 조회는 최신 활성 리비전을 반환한다.
    - 리포트 기준 거래일과 분석 범위를 함께 제공한다.
    - 리포트가 참조하는 계산 버전을 내부 조회 기준으로 고정한다.
    - 시장 개요에서 사용할 KOSPI·KOSDAQ 지수 통계와 분석 종목 기준 스캐너 통계를 함께 제공한다.
    - 시장 전체 AI 요약 상태와 내용을 함께 제공한다.
    - 당일 장 마감 리포트가 아직 공개되지 않은 경우 사용자에게 표시할 상태를 구분할 수 있게 한다.
    - 당일 장 마감 리포트가 생성 지연 상태이면 직전 활성 리포트와 생성 지연 상태를 함께 제공한다.

16. [server] 과거 장 마감 리포트 조회 API를 제공한다.
    - 서비스 운영 시작일 이후 거래일별 리포트를 조회한다.
    - 선택한 거래일의 최종 활성 리비전을 기준으로 리포트 기준 거래일, 분석 범위, 스캐너 통계, AI 요약 상태와 내용을 제공한다.
    - 선택한 거래일의 최종 활성 리비전이 참조하는 계산 버전을 내부 조회 기준으로 고정한다.
    - 휴장일, 미생성일, 서비스 시작 전 날짜의 처리를 구분할 수 있게 한다.
    - 서비스 시작일 이전 날짜는 초기 가격 데이터가 있어도 장 마감 리포트로 반환하지 않는다.

17. [server] KOSPI와 KOSDAQ 지수 시계열 API를 제공한다.
    - KOSPI와 KOSDAQ 일봉 시계열을 반환한다.
    - 시장 개요의 지수 차트가 요청한 기간에 해당하는 지수 일봉 데이터를 조회할 수 있게 한다.
    - 리포트 조회 API의 스캐너 통계와 지수 차트용 시계열 데이터를 분리한다.

18. [server] 골든크로스 보고서 API를 제공한다.
    - 요청한 리포트 리비전 기준 골든크로스 종목 목록을 반환한다.
    - 요청 기준이 없으면 최신 활성 리비전을 기본값으로 사용한다.

    - 최근 세 거래 일봉 안에 발생한 골든크로스 신호만 보고서 목록에 포함한다.
    - 신호 발생일, 현재가, K값, D값, 업종을 제공한다.
    - 종목명, 시장, 업종은 현재 종목 메타데이터가 아니라 해당 리비전의 스냅샷 기준으로 반환한다.
    - 기본 정렬은 신호 발생일 최신순, 같은 발생일 안에서는 해당 리비전의 선정 기준 거래량 높은 순으로 제공한다.

19. [server] 전체 분석 종목 API를 제공한다.

    - 과거 거래일 조회 시 선택한 거래일의 최종 활성 리비전을 기준으로 목록을 반환한다.
    - 종목명, 시장, 업종은 현재 종목 메타데이터가 아니라 해당 리비전의 스냅샷 기준으로 반환한다.

## 3. 구현하지 않는 기능

아래 항목은 MVP 구현 이슈로 만들지 않는다. 구현 중 해당 기능이 섞이지 않았는지 검증 기준으로만 사용한다.

1. KRX 전체 종목을 기준으로 한 MVP 통계를 제공하지 않는다.
2. 골든크로스를 매수 추천 또는 상승 보장으로 표현하지 않는다.
3. Spring API 또는 프론트엔드에서 기술지표를 재계산하지 않는다.
4. 사용자에게 이전 리비전 비교 화면을 제공하지 않는다.
5. 휴장일을 실패 리포트로 생성하지 않는다.
6. AI 실패를 이유로 금융 데이터 리포트 공개를 중단하지 않는다.
7. Python 배치가 실행 중 DB 스키마를 생성하거나 변경하지 않는다.
8. MVP에서 인증 기반 개인화 기능을 제공하지 않는다.
9. MVP에서 사용자 로그인과 서버 저장 관심종목을 제공하지 않는다.
10. MVP에서 운영 백업과 다중 서버 구성의 상세 정책을 구현하지 않는다.
11. MVP에서 실시간 현재가를 제공하지 않는다.
12. MVP에서 호가 및 체결 스트림을 제공하지 않는다.
13. MVP에서 장중 분봉을 제공하지 않는다.
14. MVP에서 외국인·기관 수급 데이터를 제공하지 않는다.
15. MVP에서 네이버 가격 데이터 자동 대체를 제공하지 않는다.
16. MVP에서 RSI와 Stoch RSI 보조 지표를 제공하지 않는다.
17. MVP에서 종목별 AI 설명을 제공하지 않는다.
18. MVP에서 출시 이전 일별 리포트를 소급 생성하지 않는다.

## 4. 추후 확장 기능 이슈

아래 항목은 MVP 구현 이슈에는 포함하지 않고, 추후 확장 기능으로 별도 백로그에서 관리한다.

1. 분석 종목 수 확대
2. KRX 전체 종목 분석
3. Stoch MACD 골든크로스 외 추가 신호 또는 전략 지표 도입
4. 종목 상세 보조 지표 외 추가 기술지표 도입
5. 계산 버전 v2 이상 추가
6. 종목별 AI 설명
7. 별도 관심종목 화면
8. 로그인 기반 관심종목
9. 기기 간 관심종목 동기화
10. 외국인 및 기관 수급 데이터 연동
11. 대체 데이터 소스 fallback
12. 실시간 현재가와 장중 데이터
13. 실시간 호가와 체결 스트림
14. 분봉 차트
15. 과거 리포트 소급 생성
16. 이전 리비전 비교 및 변경 원인 조회
17. 제한 병렬 수집
18. 운영 모니터링 고도화
19. 외부 백업과 보존 정책
20. 다중 서버 또는 분리 배포
21. API 계약 버전 관리
22. 상세 성능 최적화와 캐시 전략
