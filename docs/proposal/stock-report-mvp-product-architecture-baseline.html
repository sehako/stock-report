핵심 원칙

구현 계획과 이슈 분해 문서가 충돌하면 이 기준서를 우선합니다.

제품 기준

  • FinanceDataReader 단일 수집 경로로 MVP 복잡도를 제한합니다.
  • 실시간 서비스가 아니라 매 거래일 19시에 생성하는 장 마감 리포트입니다.
  • 기존 Python 스캐너의 Stoch MACD 신호 계산 방식을 재현합니다.
  • 한 종목의 실패가 전체 리포트를 막지 않도록 종목 단위로 장애를 격리합니다.

책임 경계

  • Python은 금융 데이터 생산과 분석을 담당합니다.
  • Spring은 조회 API와 사용자 기능을 담당합니다.
  • 프론트엔드는 표시와 탐색에 집중하며 기술지표를 재계산하지 않습니다.
  • AI는 계산된 사실을 설명할 뿐 수치나 원인을 새로 만들지 않습니다.

데이터 기준

MVP는 FDR의 KRX 수정주가 OHLCV를 일관되게 사용하며 원시주가와 혼합하지 않습니다.

종목 목록

StockListing("KRX")로 종목 목록과 당일 거래량을 조회합니다.

업종 정보

StockListing("KRX-DESC")가 제공하는 거래소 업종 분류를 사용합니다.

가격 이력

DataReader("KRX:종목코드", 시작일, 종료일)로 종목 가격 이력을 조회합니다.

  • 해당 거래일 KRX 종목 목록에서 종가가 1,000원 이상인 종목만 남깁니다.
  • 거래량 내림차순 상위 200개를 분석 종목으로 선정합니다.
  • 거래량에 따라 구성 종목은 매 거래일 달라질 수 있습니다.
  • 가격 행의 고유 기준은 (종목, 거래일)이며 재수집은 upsert로 반영합니다.
  • 최초 분석 진입 종목은 최근 3년 일봉을 적재합니다.
  • 기존 분석 종목은 매일 최근 10일을 재조회하고 upsert합니다.
  • 분석 대상에서 빠진 종목의 기존 데이터는 보존합니다.
  • 과거 일봉과 지표는 차트와 신호 맥락을 위한 것이며 출시 이전 리포트는 소급 생성하지 않습니다.

배치 흐름

휴장일은 실패가 아니며, 정상 종료 후 직전 리포트를 유지합니다.

1거래일 확인

KRX 거래일이 아니면 새 리포트를 만들지 않습니다.

2대상 선정

종가 1,000원 이상 종목 중 거래량 상위 200개를 선정합니다.

3순차 수집

200개 종목을 한 번에 요청하지 않고 종목별로 처리합니다.

4지표·신호 계산

MACD, Stoch MACD, 이동평균과 골든크로스를 계산합니다.

5리비전 공개

최초 리비전 공개 후 실패 종목 재시도 결과를 최종 리비전으로 반영합니다.

요청 제어

  • 종목별 요청 제한시간은 30초입니다.
  • 5개 종목이 연속으로 시간 초과하면 공급자 장애로 보고 회차를 중단합니다.
  • 실패·미처리 종목은 10분 간격으로 최대 3회 추가 재시도합니다.

게시 정책

  • 첫 수집 회차가 끝나면 정상 종목 결과를 최초 리비전으로 공개합니다.
  • 재시도 성공 때마다 새 리비전을 만들지 않습니다.
  • 모든 재시도 이후 변경분을 모아 최종 리비전 한 번만 추가합니다.

상태 모델

리포트 공개 상태, 종목 처리 상태, AI 요약 상태는 서로 다른 축입니다.

검색어를 입력하면 표 행이 필터링됩니다.

분석 중 완료 분석 생성 지연

기술지표와 신호 계산

모든 파생값, 기술지표, 신호, 순위와 업종 집계는 Python 결정론적 코드가 계산합니다.

MACD

fast 12, slow 26, signal 9 기준입니다.

Stoch MACD

K 14, D 3, smooth K 3 기준이며 MACD 선 자체를 high·low·close 입력으로 사용합니다.

이동평균

거래일 기준 5·20·60·120·240일이며 화면 라벨은 1주·1개월·3개월·6개월·1년입니다.

신호 저장과 계산 버전 기준
  • 최근 세 거래 일봉의 K·D 교차를 검사합니다.
  • 신호 식별자는 (종목, 신호 종류, 발생일, 계산 버전)입니다.
  • 동일 신호는 한 번만 저장하고 세 거래 일봉 동안 최근 신호로 재사용합니다.
  • 최초 계산법은 stoch-macd-v1로 고정합니다.
  • 계산법 변경 시 기존 값을 덮어쓰지 않고 v2 결과를 추가합니다.

업종 분석 규칙

업종 분석은 해당 거래일 분석 종목 200개만 대상으로 하며, 실제 사용한 기준을 리비전별로 저장·노출합니다.

면적 기준 fallback

트리맵 면적은 업종별 시가총액 합계를 기본으로 사용하고, 불안정하면 거래대금 합계, 그마저 불안정하면 분석 종목 수를 같은 리비전 전체에 적용합니다.

market_cap trade_value stock_count

색상 기준

트리맵 색상은 업종 종목의 일간 등락률 단순 평균입니다. 골든크로스 스캐너의 개별 종목 중심 관점에 맞추기 위해 단순 평균을 사용합니다.

신호 비율

최근 골든크로스 비율은 최근 세 거래 일봉 안에 신호가 발생한 종목 수를 신호 판정 완료 종목 수로 나눈 값입니다.

업종 통계 저장·노출 기준
  • 업종별 포함 종목 수와 최근 골든크로스 종목 수를 함께 표시합니다.
  • 실제 사용한 면적 기준은 market_cap, trade_value, stock_count 중 하나로 저장하고 API 응답에 노출합니다.
  • 분석 데이터 부족, 데이터 갱신 실패, 당일 거래 없음 종목은 신호 비율 분모에서 제외합니다.
  • 분모 제외 종목 수는 업종 분석 데이터와 API 응답에서 함께 표시합니다.

AI 시장 요약

AI는 거래일별 시장 전체 요약 한 건으로 제한하며, 금융 데이터 리비전과 별도 수명주기로 관리합니다.

입력과 범위

  • 종목별 AI 설명은 생성하지 않습니다.
  • KOSPI·KOSDAQ 지수 통계와 거래량 상위 200개 스캐너 통계를 분리해 제공합니다.
  • 입력 예시는 지수 등락, 상승·하락 종목 수, 최근 골든크로스 수, 주요 업종 통계입니다.
  • AI 요약은 참조한 리포트 리비전과 입력 해시를 함께 저장합니다.

실행과 실패

  • 종목 데이터 재시도가 모두 끝난 뒤 AI를 한 번 생성합니다.
  • AI 실패 시 1분, 5분, 15분 간격으로 최대 3회 재시도합니다.
  • 재시도 중에는 분석 중, 모두 실패하면 분석 생성 지연으로 표시합니다.
  • AI 상태와 내용 변경만으로 금융 데이터 리비전을 추가하거나 활성 리비전을 교체하지 않습니다.

시스템 아키텍처

Python Batch Worker와 Spring API Server가 PostgreSQL을 공유하되 책임을 분리합니다.

Frontend

시장 개요, 골든크로스, 전체 종목, 종목 상세, 업종, 과거 리포트 화면을 제공합니다.

Spring API Server

조회 전용 GET API와 애플리케이션 로직을 담당하며 지표를 재계산하지 않습니다.

Python Batch Worker

종목 선정, 수집, upsert, 지표, 신호, 업종 집계, AI 입력 생성을 담당합니다.

조회 API와 공유 데이터베이스를 기준으로 연결
FinanceDataReader

KRX 종목 목록, 설명 목록, 종목 가격 이력, 시장 지수를 조회합니다.

PostgreSQL

활성 리비전이 사용자에게 보이는 단일 기준이며 최신 적재 행을 직접 노출하지 않습니다.

Flyway

DB 구조 변경은 Spring의 Flyway가 단독 관리하고 Python은 승인된 테이블만 사용합니다.

  • 가격 일봉은 리비전마다 복제하지 않습니다.
  • 사용자에게는 거래일별 활성 리비전 하나만 노출합니다.
  • AI 요약은 리비전을 참조하지만 금융 데이터 리비전의 일부로 보지 않습니다.

API와 화면 표시 원칙

공개 API는 latest 또는 tradeDate를 입력으로 받고 Spring이 활성 리비전을 해석합니다.

조회 API 개념

  • 최신 장 마감 리포트와 거래일별 과거 리포트
  • 활성 리비전 기준 골든크로스 종목 목록
  • 활성 리비전 기준 전체 분석 종목 목록
  • 종목 가격·지표·신호 시계열과 업종 분석
  • KOSPI·KOSDAQ 지수 시계열

사용자 표시 원칙

  • 모든 화면에 데이터 기준 거래일을 표시합니다.
  • 분석 범위가 KRX 전체가 아니라 거래량 상위 200개임을 명시합니다.
  • 과거 리포트 상세 차트는 선택 거래일 이후 데이터를 섞지 않습니다.
  • 신호 발생일과 보고서 표시일을 구분합니다.

검증과 수용 기준

데이터 수집, 분석, 장애 격리, API와 화면 기준으로 MVP 완료 여부를 판단합니다.

국내 종목 가격 호출은 모두 KRX: 소스를 명시합니다.
동일 종목·거래일 재실행 시 가격 중복 행이 생기지 않습니다.
신규 분석 종목은 지표 계산에 필요한 과거 이력을 확보합니다.
휴장일은 실패 리포트를 만들지 않습니다.
고정 가격 fixture에서 MACD와 Stoch MACD 결과가 반복 실행마다 동일합니다.
K선이 D선을 아래에서 위로 돌파한 날짜만 신호로 저장합니다.
동일 신호를 여러 보고서가 참조해도 signal_event는 한 건입니다.
계산 버전이 다른 지표가 한 리포트에 섞이지 않습니다.
한 종목 수집 실패가 다른 종목 처리를 멈추지 않습니다.
데이터 준비 중과 데이터 갱신 실패가 구분됩니다.
종목 목록 실패 시 직전 활성 리포트를 유지합니다.
AI 실패 시에도 계산 결과와 차트를 제공합니다.
AI 상태와 내용 변경만으로 새 금융 데이터 리비전이 생성되지 않습니다.
Spring은 활성 리비전만 기본 조회합니다.
응답에는 거래일, 활성 리비전 번호, 계산 버전, 공개 상태, AI 상태가 포함됩니다.
상세 차트와 지표 시계열의 마지막 날짜가 요청 기준일을 넘지 않습니다.
업종 통계는 신호 판정 완료 종목만 비율 분모로 사용합니다.
업종 트리맵 응답은 실제 사용한 면적 기준과 색상 기준을 구분해 제공합니다.

최종 결정 요약

MVP는 실시간 종합 주식 플랫폼이 아니라 FDR의 KRX 일봉을 기반으로 거래량 상위 200개를 안정적으로 스캔하는 장 마감 리포트입니다. Python이 금융 데이터와 분석 결과를 생산하고 Spring이 게시 완료된 결과를 조회 전용 API로 제공합니다. PostgreSQL의 활성 리비전이 사용자에게 보이는 단일 기준이며, 한 종목의 실패와 AI 실패는 정상 결과로부터 격리합니다.
