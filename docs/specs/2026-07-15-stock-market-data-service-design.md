# 주식 시장 데이터 조회 서비스 MVP 설계

## 목적

이 문서는 코스피/코스닥 지수와 국내 주식 종목 데이터를 제공하는 웹 애플리케이션의 1차 MVP 범위와 기술 설계를 정의한다.

MVP의 목표는 실시간 투자 서비스가 아니라, 장 마감 후 수집한 일봉 데이터를 기반으로 사용자가 시장 지수와 주요 종목 데이터를 안정적으로 조회할 수 있게 하는 것이다.

## MVP 범위

### 포함 기능

- 코스피/코스닥 최신 수치 조회
- 코스피/코스닥 1개월, 3개월, 1년 일봉 차트 조회
- 거래량 상위 200개 KRX 종목 목록 조회
- 종목명 또는 종목코드 검색
- 시장 구분 필터
- 최신 종가, 등락률, 거래량 기준 정렬
- 페이지네이션
- 종목별 상세 정보 조회
- 종목별 1개월, 3개월, 1년 일봉 차트 조회
- 장 마감 후 데이터 수집 배치
- 특정 종목, 특정 지수, 특정 기간 재수집

### 제외 기능

- 실시간 스트리밍 시세
- 관심 종목 관리
- 사용자 계정
- 투자 리포트 또는 AI 분석 콘텐츠
- 전체 KRX 종목 적재
- 배치 실패 이력 관리 화면
- 관리자 화면

## 시스템 구조

시스템은 Python 배치, Spring Boot 백엔드, React 프론트엔드로 분리한다.

### Python 배치

Python 배치는 `FinanceDataReader`를 사용해 장 마감 후 시장 데이터를 수집한다.

- `FinanceDataReader.StockListing("KRX")`로 KRX 종목 목록을 조회한다.
- 거래량 기준 상위 200개 종목을 선정한다.
- KRX 제공 목록을 그대로 사용하며, ETF, ETN, 우선주 등 비보통주를 별도로 제외하지 않는다.
- 선정된 종목의 조회 가능한 최초 일봉부터 최신 일봉까지 RDB에 저장한다.
- 코스피와 코스닥 지수의 조회 가능한 최초 일봉부터 최신 일봉까지 RDB에 저장한다.
- 이후 실행부터는 마지막 적재일 이후 데이터만 추가 적재한다.

### Spring Boot 백엔드

Spring Boot 백엔드는 데이터 수집을 직접 수행하지 않고 RDB에 저장된 데이터를 조회하는 API를 제공한다.

- 종목 목록 API
- 종목 상세 API
- 종목 일봉 차트 API
- 지수 최신 수치 API
- 지수 일봉 차트 API

패키지 구조는 `docs/architecture/backend.md`의 도메인 중심 계층 구조를 따른다.

### React 프론트엔드

React 프론트엔드는 사용자가 시장 데이터와 종목 데이터를 조회하는 화면을 제공한다.

- 시장 개요 화면
- 종목 목록 화면
- 종목 상세 화면

디렉터리 구조와 컴포넌트 분리는 `docs/architecture/frontend.md`의 기능 중심 구조를 따른다.

## 데이터 저장 방식

종목 메타데이터, 종목 일봉, 지수 일봉은 모두 RDB에 저장한다.

1차 MVP의 데이터 규모는 RDB 저장에 적합하다. 거래량 상위 200개 종목의 장기 일봉 데이터는 수십만에서 백만 row 수준으로 예상되며, 일반적인 PostgreSQL 또는 MySQL에서 충분히 처리 가능하다.

파일 시스템에 시계열 데이터를 저장하는 방식은 초기에는 단순해 보일 수 있으나, Spring API에서 검색, 정렬, 기간 필터링, 중복 방지, 테스트를 처리하기 복잡해질 수 있으므로 MVP 기본안에서는 제외한다.

## 데이터 모델

### `stock`

종목 메타데이터를 저장한다.

| 컬럼 | 설명 |
| --- | --- |
| `id` | 내부 기본키 |
| `stock_code` | 종목코드 |
| `stock_name` | 종목명 |
| `market` | 시장 구분 |
| `tracked` | 현재 거래량 상위 200개 포함 여부 |

`tracked`는 현재 MVP 조회 대상 여부를 나타낸다. 이전 배치에서 상위 200개였지만 최신 배치에서 제외된 종목은 `tracked = false`로 갱신한다.

`market`, `stock_code` 조합에 유니크 제약을 둔다. 외부 데이터와 API에서는 `market`, `stock_code` 조합으로 종목을 식별하고, 내부 가격 테이블에서는 `id`를 참조한다.

마지막 적재일은 `stock`에 별도 컬럼으로 저장하지 않는다. 배치는 `stock_price`에서 해당 종목의 `MAX(trade_date)`를 조회해 마지막 적재일을 계산한다.

### `stock_price`

종목별 일봉 가격 데이터를 저장한다.

| 컬럼 | 설명 |
| --- | --- |
| `id` | 내부 기본키 |
| `stock_id` | `stock.id`를 참조하는 내부 종목 식별자 |
| `trade_date` | 거래일 |
| `open_price` | 시가 |
| `high_price` | 고가 |
| `low_price` | 저가 |
| `close_price` | 종가 |
| `volume` | 거래량 |
| `change_rate` | 등락률 |

`stock_id`, `trade_date` 조합에 유니크 제약을 둔다. 배치를 반복 실행해도 같은 종목의 같은 거래일 데이터가 중복 저장되지 않게 한다.

배치는 종목 일봉을 저장하기 전에 `stock` 테이블에서 `market`, `stock_code` 조합으로 `stock.id`를 조회한 뒤 `stock_price.stock_id`에 저장한다.

### `market_index_price`

코스피/코스닥 지수의 일봉 데이터를 저장한다.

| 컬럼 | 설명 |
| --- | --- |
| `id` | 내부 기본키 |
| `index_code` | 내부 지수 코드. `KOSPI`, `KOSDAQ` |
| `trade_date` | 거래일 |
| `open_price` | 시가 |
| `high_price` | 고가 |
| `low_price` | 저가 |
| `close_price` | 종가 |
| `volume` | 거래량 |
| `change_rate` | 등락률 |

테이블은 코스피와 코스닥으로 나누지 않고, `index_code`로 구분한다. `index_code`, `trade_date` 조합에 유니크 제약을 둔다.

`index_code`는 `KOSPI`, `KOSDAQ`만 허용한다.

FDR의 실제 지수 심볼은 배치 내부에서 내부 코드로 매핑한다. 애플리케이션 내부와 API에서는 `KOSPI`, `KOSDAQ`을 사용한다.

## API 설계

### 지수 최신 수치 조회

```http
GET /api/market-indexes
```

코스피와 코스닥의 최신 수치, 전일 대비 값, 등락률을 반환한다.

### 지수 차트 조회

```http
GET /api/market-indexes/{indexCode}/prices?period=1M|3M|1Y
```

지정한 지수의 기간별 일봉 데이터를 반환한다.

### 종목 목록 조회

```http
GET /api/stocks?market=&keyword=&sort=&page=&size=
```

거래량 상위 200개 대상 종목 목록을 조회한다.

- `market`: 시장 구분 필터
- `keyword`: 종목명 또는 종목코드 검색어
- `sort`: 최신 종가, 등락률, 거래량 기준 정렬
- `page`, `size`: 페이지네이션

응답에는 종목코드, 종목명, 시장 구분, 최신 종가, 등락률, 거래량, 마지막 거래일을 포함한다.

### 종목 상세 조회

```http
GET /api/stocks/{stockCode}
```

종목 기본 정보와 최신 일봉 데이터를 반환한다.

### 종목 차트 조회

```http
GET /api/stocks/{stockCode}/prices?period=1M|3M|1Y
```

지정한 종목의 기간별 일봉 데이터를 반환한다.

## 배치 설계

### 기본 실행

기본 배치는 전체 MVP 조회 대상을 갱신한다.

1. KRX 전체 종목 목록을 조회한다.
2. 거래량 기준 상위 200개 종목을 선정한다.
3. `stock` 테이블을 upsert한다.
4. 현재 상위 200개 종목은 `tracked = true`로 갱신한다.
5. 이전에 저장되었지만 현재 상위 200개에서 빠진 종목은 `tracked = false`로 갱신한다.
6. 각 tracked 종목의 마지막 적재일을 `stock_price`의 최대 거래일로 계산하고, 그 이후 일봉을 조회해 `stock_price`에 `stock_id`, `trade_date` 기준으로 upsert한다.
7. 코스피/코스닥 지수의 마지막 적재일을 `market_index_price`의 최대 거래일로 계산하고, 그 이후 일봉을 조회해 `market_index_price`에 `index_code`, `trade_date` 기준으로 upsert한다.
8. 성공, 스킵, 실패 결과를 로그로 남긴다.

### 지정 실행

배치는 실패 복구와 데이터 보정을 위해 실행 옵션을 제공한다.

```bash
python batch.py --stock-code 005930
python batch.py --index-code KOSPI
python batch.py --start-date 2024-01-01 --end-date 2024-12-31
python batch.py --stock-code 005930 --start-date 2024-01-01 --end-date 2024-01-31
python batch.py --index-code KOSDAQ --start-date 2024-01-01 --end-date 2024-01-31
```

옵션을 지정하지 않으면 기본 실행으로 동작한다.

### 실패 처리

- 한 종목 수집 실패가 전체 배치를 중단하지 않는다.
- 실패 종목 코드, 종목명, 오류 메시지를 로그에 남긴다.
- 지수 수집 실패 시 지수 코드와 오류 메시지를 로그에 남긴다.
- 이미 적재된 데이터는 유니크 제약과 upsert로 중복 저장을 방지한다.
- 실패 로그를 보고 특정 종목, 특정 지수, 특정 기간만 재실행할 수 있다.
- 배치 실행 이력 테이블은 MVP에서는 만들지 않고 로그 기반으로 시작한다.

## 오류 처리

### API 오류

- 존재하지 않는 종목코드 요청은 404 응답으로 처리한다.
- 존재하지 않는 지수 코드 요청은 404 응답으로 처리한다.
- 지원하지 않는 기간 값은 400 응답으로 처리한다.
- 데이터가 아직 적재되지 않은 경우 빈 목록 또는 명확한 상태 응답을 반환한다.

### 화면 오류

- API 오류가 발생하면 화면 영역별 오류 상태를 표시한다.
- 데이터가 없는 경우 빈 상태를 표시한다.
- 차트 기간 변경 중에는 로딩 상태를 표시한다.

## 결정 사항 요약

- MVP는 시장 데이터 조회 서비스로 제한한다.
- 데이터는 실시간 스트리밍이 아니라 장 마감 후 일봉 기준으로 제공한다.
- 수집은 Python 배치와 `FinanceDataReader`가 담당한다.
- 조회 대상 종목은 KRX 거래량 상위 200개로 제한한다.
- KRX 제공 목록을 그대로 사용하며 ETF, ETN, 우선주를 별도 제외하지 않는다.
- 종목 메타데이터, 종목 일봉, 지수 일봉은 모두 RDB에 저장한다.
- Spring Boot는 RDB 조회 API만 제공한다.
- React는 시장 개요, 종목 목록, 종목 상세 화면을 제공한다.
- 배치는 전체 실행과 지정 재수집을 모두 지원한다.
