# Python 배치의 재시도 정합성 보장

Python 배치는 Spring Batch를 사용하지 않고, PostgreSQL의 고유키, upsert, 트랜잭션, 실행 상태 테이블 및 advisory lock을 조합해 재시도 정합성을 보장한다. Python 기술 스택은 배치 실행과 데이터 계산에 집중하고, 재시도 후 중복 데이터 방지와 공개 리비전의 원자성은 데이터베이스 제약과 명시적인 상태 전이로 관리한다.

Spring Batch를 도입하는 대안은 JobRepository와 재시작 모델을 기본 제공하지만, 현재 아키텍처에서 Python이 금융 데이터 수집과 기술지표 계산을 담당한다는 결정과 충돌한다. Python에 별도 배치 프레임워크를 도입하는 대안은 MVP의 단일 서버, 하루 1회 장 마감 배치에 비해 운영 복잡도가 크므로 채택하지 않는다.

## 결정 사항

- Python 배치는 `Python 3.12`, `uv`, `FinanceDataReader`, `pandas`, `numpy`, `psycopg 3`, `SQLAlchemy Core`, `pydantic-settings`, `pytest`를 기본 스택으로 사용한다.
- 컨테이너 내부 스케줄링은 `cron` 또는 `supercronic`으로 수행하되, 중복 실행 방지는 스케줄러가 아니라 PostgreSQL advisory lock으로 보장한다.
- 원본 일봉, 기술지표, 신호, 종목 처리 상태는 고유키 기반 `INSERT ... ON CONFLICT`로 기록한다.
- 전체 배치를 하나의 장기 트랜잭션으로 묶지 않고, 종목 단위의 짧은 트랜잭션으로 처리한다.
- 배치 실행 상태는 별도 테이블에 저장해 프로세스 중단 후에도 실패 종목과 재시도 대상을 복구할 수 있게 한다.
- 리포트 공개는 draft 리비전 작성 후, 활성 리비전 전환을 하나의 트랜잭션으로 처리한다.

## Consequences

- 동일 거래일 배치를 재실행해도 가격, 지표, 신호 및 종목 상태 데이터가 중복 생성되지 않는다.
- 한 종목의 실패는 해당 종목 트랜잭션만 rollback하며, 다른 종목의 처리와 최초 리비전 공개를 막지 않는다.
- 프로세스 중단 후 재실행 시 실행 상태 테이블을 기준으로 미완료 또는 재시도 가능 종목만 다시 처리할 수 있다.
- 같은 거래일 배치가 동시에 실행되는 상황은 advisory lock으로 차단한다.
- 사용자에게 노출되는 활성 리비전은 거래일별 하나만 유지해야 하며, 이를 데이터베이스 제약과 publish 트랜잭션으로 강제한다.
- Python은 Flyway가 생성한 테이블만 사용하고, 스키마 생성이나 변경을 수행하지 않는다.
- Spring Batch 수준의 정합성은 프레임워크 기능이 아니라 테이블 설계, 제약조건, 상태 전이 테스트로 검증해야 한다.
