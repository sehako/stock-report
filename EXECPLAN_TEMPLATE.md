# <짧고 행동 지향적인 설명>

이 ExecPlan은 살아 있는 문서다. 작업이 진행되는 동안 `진행 상황`, `예상 밖의 발견`, `결정 기록`, `결과와 회고` 섹션을 최신 상태로 유지해야 한다.

이 문서는 저장소 루트의 `PLANS.md`를 따른다. 이 계획을 수정할 때는 `PLANS.md`의 요구사항과 지침을 함께 확인한다.

## 목적과 큰 그림

이 변경 후 사용자가 무엇을 할 수 있게 되는지 몇 문장으로 설명한다. 이전에는 할 수 없었지만 변경 후 가능해지는 사용자 관찰 가능 동작을 적는다. 구현이 끝난 뒤 그 동작을 어떻게 직접 확인할 수 있는지도 함께 적는다.

## 진행 상황

세부 단계를 체크박스로 추적한다. 모든 중단 지점은 여기에 문서화한다. 부분적으로 완료된 작업은 완료된 부분과 남은 부분이 드러나도록 항목을 나누거나 설명을 덧붙인다. 타임스탬프는 진행 속도와 작업 이력을 확인할 수 있게 작성한다.

- [ ] (YYYY-MM-DD HH:MMZ) 계획 초안 작성.
- [ ] 저장소 구조와 관련 코드 조사.
- [ ] 구현 위치와 검증 방법 확정.
- [ ] 구현 완료.
- [ ] 검증 완료.

## 예상 밖의 발견

구현 중 발견한 예상치 못한 동작, 버그, 제약, 성능 특성, 설계상 통찰을 기록한다. 가능한 경우 짧은 터미널 출력, 테스트 결과, 코드 위치 같은 증거를 함께 남긴다.

- 관찰: 아직 없음.
  증거: 아직 없음.

## 결정 기록

계획을 작업하면서 내린 모든 중요한 결정을 기록한다. 결정은 무엇을 선택했는지뿐 아니라 왜 그렇게 했는지까지 설명해야 한다.

- 결정: 아직 없음.
  근거: 아직 없음.
  날짜/작성자: YYYY-MM-DD / 작성자

## 결과와 회고

주요 마일스톤을 완료했거나 전체 작업을 마쳤을 때 결과를 요약한다. 원래 목적과 비교해 무엇이 달성되었는지, 무엇이 남았는지, 다음 작업자가 알아야 할 교훈이 무엇인지 적는다.

아직 작업이 완료되지 않았다.

## 맥락과 방향 안내

이 작업과 관련된 현재 저장소 상태를 초보자 기준으로 설명한다. 관련 파일은 저장소 루트 기준 전체 상대 경로로 적는다. 사용할 모듈, 함수, 클래스, 명령을 정확히 이름 붙인다. 비자명한 용어는 처음 등장할 때 쉬운 말로 정의한다.

## 작업 계획

구현 순서를 산문으로 설명한다. 각 단계에서 어떤 파일의 어떤 위치를 바꾸는지, 어떤 새 파일을 만드는지, 어떤 기존 동작과 연결되는지 구체적으로 적는다. 모호한 선택지는 여기서 결론을 내리고 이유를 함께 적는다.

## 마일스톤

각 마일스톤은 독립적으로 검증 가능해야 한다. 마일스톤마다 끝났을 때 새로 존재하는 동작, 실행할 명령, 관찰해야 할 결과를 설명한다.

첫 번째 마일스톤은 관련 코드와 실행 경로를 조사해 구현 위치와 검증 방법을 확정하는 것이다. 이 마일스톤이 끝나면 계획의 `맥락과 방향 안내`, `작업 계획`, `검증과 수락 기준`이 실제 저장소 구조에 맞게 갱신되어 있어야 한다.

두 번째 마일스톤은 최소 동작을 구현하고 자동 검증을 추가하는 것이다. 이 마일스톤이 끝나면 변경 전 실패하고 변경 후 통과하는 테스트 또는 사람이 재현할 수 있는 검증 절차가 있어야 한다.

세 번째 마일스톤은 마무리 검증과 문서 갱신이다. 이 마일스톤이 끝나면 모든 관련 테스트가 통과하고, `결과와 회고`에 결과와 남은 위험이 정리되어 있어야 한다.

## 구체적인 단계

실행할 정확한 명령과 실행 위치를 적는다. 명령이 출력을 만든다면 성공을 판단할 수 있는 짧은 예상 출력을 함께 적는다. 작업이 진행되면서 실제 명령과 결과에 맞게 이 섹션을 갱신한다.

작업 디렉터리:

    /Users/sehako/workspace/stock-report

예상 명령:

    <프로젝트에 맞는 조사, 빌드, 테스트 명령을 여기에 적는다>

## 검증과 수락 기준

구현이 완료되었는지 확인하는 방법을 사용자 관찰 가능 동작으로 적는다. 테스트가 있다면 정확한 테스트 명령과 기대 결과를 적는다. 수락 기준은 내부 구현 세부사항이 아니라 사람이 직접 확인할 수 있는 입력과 출력으로 표현한다.

수락 기준:

- 사용자는 <구체적인 입력 또는 행동>을 수행할 수 있다.
- 시스템은 <구체적인 출력 또는 상태 변화>를 보여준다.
- `<테스트 명령>`을 실행하면 <기대 결과>가 나온다.

## 멱등성과 복구

각 단계가 여러 번 실행되어도 안전한지 설명한다. 실패했을 때 다시 시도하는 방법, 되돌릴 수 있는 방법, 임시 파일이나 생성물을 정리하는 방법을 적는다. 파괴적인 작업이 필요하다면 백업 또는 안전한 대안을 먼저 명시한다.

## 산출물과 메모

성공을 증명하는 중요한 출력, 짧은 diff, 로그, 관찰 내용을 보관한다. 큰 코드 덩어리 대신 재현에 필요한 핵심 조각만 들여쓰기 된 예시로 포함한다.

## 인터페이스와 의존성

새로 만들거나 변경해야 하는 인터페이스, 타입, 함수, 모듈, 외부 의존성을 명확히 적는다. 이름은 안정적인 경로로 작성한다. 예를 들어 `src/main/kotlin/...` 파일의 함수명이나 `package.module.Interface`처럼 실제 저장소에서 찾을 수 있는 이름을 사용한다.

계획 변경 메모:

- YYYY-MM-DD / 작성자: 계획 초안 작성. 이유: 작업을 자기완결적으로 수행하기 위한 초기 실행 계획을 마련하기 위해서.
