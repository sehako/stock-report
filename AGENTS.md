# 핵심 원칙

- 사용자의 명시적인 구현 지시 없이는 코드베이스를 수정하지 않는다.
- 요청받은 범위를 벗어난 리팩터링이나 구조 변경을 수행하지 않는다.
- 기존 코드와 문서의 설계 및 스타일을 우선하여 따른다.
- 모든 마크다운 문서는 한글로 작성한다.
- 코드, 클래스명, 함수명, 라이브러리명 등 기술 식별자는 원래 표기를 유지한다.
- 커밋 메시지는 `type(scope): description` 형식을 따른다.
  - `description`은 한글로 작성한다.
  - body와 footer는 작성하지 않는다.
- 원격 이슈 작성 시 `.github/ISSUE_TEMPLATE`에 있는 문서를 기반으로 작성한다.

# 프로젝트 구조

- `docs`: 기획서, 아키텍처, 구현 계획서 등 문서 산출물
- `app`: 애플리케이션 코드 디렉터리
  - `backend`: Kotlin + Spring Boot
  - `frontend`: TypeScript + React.js

# 코드 작성 규칙

- 코드 작성 전 작업 대상과 관련된 기존 코드 및 문서를 확인한다.
- 코드 작성 시 다음 아키텍처 문서를 따른다.
  - `app/backend`: `docs/architecture/backend.md`
  - `app/frontend`: `docs/architecture/frontend.md`

- 기존 코드의 네이밍, 디렉터리 구조, 예외 처리 방식 및 응답 형식을 유지한다.
- 요청과 직접 관련 없는 코드 정리나 포맷 변경은 수행하지 않는다.
- 새로운 의존성은 기존 의존성만으로 구현하기 어려운 경우에만 추가한다.
- 비밀번호, API 키, 토큰 등 민감정보를 코드나 문서에 직접 작성하지 않는다.

# 검증 규칙

- 코드 수정 후 가능한 범위에서 관련 테스트, 빌드 또는 정적 검사를 실행한다.
- 새로운 동작을 추가하거나 결함을 수정한 경우 가능한 한 관련 테스트를 작성한다.
- 검증에 실패하거나 검증하지 못한 항목이 있으면 결과에 명시한다.
- 데이터 삭제, 강제 초기화, 강제 푸시 등 복구하기 어려운 작업은 명시적인 지시 없이 수행하지 않는다.
