# 프로젝트 구조

- `client/`: TypeScript / React.js / TailwindCSS
- `server/`: Kotlin / Spring Boot / JPA
- `worker/`: Python / FinanceDataReader / pandas / SQLAlchemy Core
- `docs/`: 기획서, 코드베이스 조사, 구현 계획, API 문서, 목업 등 프로젝트 문서 산출물

# 핵심 원칙

- 사용자의 명시적인 구현 지시 없이는 코드베이스를 수정하지 않는다.
- 조사, 계획, 리뷰 단계에서는 파일을 수정하지 않는다.
- 사용자가 승인한 구현 계획이 있는 경우에만 구현을 수행한다.
- 구현 중 계획 밖 변경이 필요하면 임의로 수정하지 않고 사용자에게 먼저 보고한다.
- API 스펙, DB 스키마, 외부 의존성, 프로젝트 구조 변경은 사용자의 명시적인 승인 없이 수행하지 않는다.
- 모든 문서 산출물은 한글로 작성한다.
- 커밋 메시지는 `type(scope): description` 형식을 따른다.
- 커밋 메시지의 `description`은 한글로 작성하고, body나 footer는 작성하지 않는다.

# 작업 흐름

- 작업은 사용자의 지시에 따라 조사, 계획, 구현 및 리뷰로 나누어 진행한다.
- 각 단계는 사용자의 명시적인 지시가 있을 때만 수행한다.

## 1. 조사 단계

- 사용자가 `조사해줘`라고 지시하면 조사를 진행한다.
- 현재 코드 구조, 관련 파일, 기존 구현 패턴을 파악한다.
- 도메인 흐름, 트랜잭션 경계, 예외 처리 방식, 테스트 구조를 확인한다.
- 코드 수정은 하지 않는다.
- 조사 결과는 조사한 코드베이스에 따라 다음 경로에 정리한다.
  - `docs/research/client`
  - `docs/research/server`
  - `docs/research/worker`

## 2. 계획 단계

- 사용자가 `계획 세워줘`라고 지시하면 계획을 진행한다.
- 계획 과정에서는 `grill-with-docs` 스킬을 사용한다.
- 수정할 파일, 파일별 변경 내용, 테스트 범위, 변경 금지 범위를 정리한다.
- 구현 완료 조건을 명확히 작성한다.
- 사용자가 명시적으로 구현을 요청하기 전까지 코드를 수정하지 않는다.
- 계획은 구현 계획 유형에 따라 다음 경로에 정리한다.
  - `docs/implementation/client`
  - `docs/implementation/server`
  - `docs/implementation/worker`

- 계획 단계의 산출물은 구현 세션의 기준 문서로 사용한다.

## 3. 구현 단계

- 사용자가 명시적으로 구현을 요청하면 구현을 진행한다.
- 승인된 구현 계획에 따라 코드를 수정한다.
- 계획에 없는 파일은 임의로 수정하지 않는다.
- API 스펙, DB 스키마, 외부 의존성은 임의로 변경하지 않는다.
- 계획 밖 변경이 필요하다고 판단되면 구현을 중단하고 사용자에게 보고한다.
- 구현 후 가능한 테스트를 실행한다.
- 테스트를 실행할 수 없는 경우, 실행하지 못한 이유를 사용자에게 보고한다.

## 4. 구현 후 리뷰 단계

- 구현 완료 후 `code-reviewer` 서브에이전트를 호출하여 변경 diff를 검토한다.
- 리뷰 기준은 승인된 구현 계획, 완료 조건, 테스트 요구사항, 변경 금지 범위로 한다.
- `code-reviewer`는 코드를 직접 수정하지 않고 문제점과 근거만 보고한다.
- 리뷰 결과는 `필수 수정`, `권장 수정`, `문제 없음`으로 구분한다.
- 계획 범위 안의 필수 수정 사항은 최대 1회까지 자동으로 수정한 뒤 관련 테스트를 다시 실행한다.
- 추가적인 설계 변경, API 변경, DB 스키마 변경, 외부 의존성 추가가 필요하면 임의로 수정하지 않고 사용자에게 보고한다.

## 5. 최종 보고

최종 응답에는 다음 내용을 포함한다.

- 변경 파일 목록
- 구현 요약
- 테스트 실행 결과
- 리뷰 결과
- 남은 리스크
- 사용자 확인이 필요한 사항
