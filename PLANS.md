# Codex 실행 계획(ExecPlans):

이 문서는 실행 계획("ExecPlan")의 요구사항을 설명한다. ExecPlan은 코딩 에이전트가 따라가며 동작하는 기능이나 시스템 변경을 제공할 수 있는 설계 문서다. 독자는 이 저장소를 전혀 모르는 완전한 초보자라고 가정한다. 독자는 현재 작업 트리와 당신이 제공하는 단 하나의 ExecPlan 파일만 가지고 있다. 이전 계획에 대한 기억도 없고 외부 맥락도 없다.

## ExecPlans와 PLANS.md 사용 방법

실행 가능한 명세(ExecPlan)를 작성할 때는 PLANS.md를 _글자 그대로_ 따른다. 문맥에 PLANS.md가 없다면 PLANS.md 파일 전체를 읽어 기억을 새로 고친다. 정확한 명세를 만들기 위해 소스 자료를 철저히 읽고, 필요하면 다시 읽는다. 새 ExecPlan 초안은 `EXECPLAN_TEMPLATE.md`를 복사해 시작하되, 템플릿의 모든 안내 문구를 실제 작업 맥락으로 교체한다. 명세를 만들 때는 템플릿에서 시작해 조사하면서 내용을 채운다.

실행 가능한 명세(ExecPlan)를 구현할 때는 사용자에게 "다음 단계"를 물어보지 않는다. 그냥 다음 마일스톤으로 진행한다. 모든 섹션을 최신 상태로 유지하고, 멈추는 지점마다 목록 항목을 추가하거나 나누어 지금까지 이룬 진행 상황과 다음 단계를 명확히 적는다. 모호함은 자율적으로 해결하고, 자주 커밋한다.

실행 가능한 명세(ExecPlan)를 논의할 때는 훗날을 위해 명세 안의 로그에 결정을 기록한다. 명세가 왜 변경되었는지 모호하지 않게 분명해야 한다. ExecPlan은 살아 있는 문서이며, 언제나 _오직_ ExecPlan만으로, 다른 작업 없이도 다시 시작할 수 있어야 한다.

까다로운 요구사항이나 큰 미지수가 있는 설계를 조사할 때는 마일스톤을 사용해 개념 증명, "장난감 구현" 등을 구현한다. 이를 통해 사용자의 제안이 실현 가능한지 검증할 수 있다. 라이브러리를 찾거나 확보해 소스 코드를 읽고, 깊이 조사하며, 더 완전한 구현을 안내할 프로토타입을 포함한다.

## 요구사항

협상 불가능한 요구사항:

- 모든 ExecPlan은 완전히 자기완결적이어야 한다. 자기완결적이라는 말은 현재 형태만으로도 초보자가 성공하는 데 필요한 모든 지식과 지시가 들어 있다는 뜻이다.
- 모든 ExecPlan은 살아 있는 문서다. 기여자는 진행이 이루어지고, 발견이 생기고, 설계 결정이 확정될 때마다 이를 수정해야 한다. 각 수정본도 완전히 자기완결적이어야 한다.
- 모든 ExecPlan은 이 저장소에 대한 사전 지식이 전혀 없는 완전한 초보자도 기능을 처음부터 끝까지 구현할 수 있게 해야 한다.
- 모든 ExecPlan은 단지 "정의를 충족하는" 코드 변경이 아니라, 명확히 시연 가능한 동작을 만들어야 한다.
- 모든 ExecPlan은 모든 전문 용어를 쉬운 말로 정의해야 하며, 정의하지 않을 용어는 사용하지 않는다.

목적과 의도가 먼저다. 이 작업이 사용자의 관점에서 왜 중요한지 몇 문장으로 설명하며 시작한다. 이 변경 후 누군가가 이전에는 할 수 없었던 무엇을 할 수 있는지, 그리고 그것이 동작하는 것을 어떻게 볼 수 있는지를 설명한다. 그다음 독자가 그 결과에 도달할 수 있도록 정확한 단계를 안내한다. 무엇을 편집할지, 무엇을 실행할지, 무엇을 관찰해야 하는지를 포함한다.

당신의 계획을 실행하는 에이전트는 파일을 나열하고, 파일을 읽고, 검색하고, 프로젝트를 실행하고, 테스트를 실행할 수 있다. 그러나 이전 맥락을 알지 못하며, 이전 마일스톤에서 당신이 무엇을 뜻했는지 추론할 수 없다. 당신이 의존하는 모든 가정을 반복해서 적는다. 외부 블로그나 문서를 가리키지 않는다. 지식이 필요하다면 계획 안에 당신의 말로 포함한다. ExecPlan이 이전 ExecPlan을 기반으로 하고 그 파일이 저장소에 체크인되어 있다면, 참조로 포함한다. 그렇지 않다면 그 계획의 관련 맥락을 모두 포함해야 한다.

## 형식

형식과 외피는 단순하고 엄격하다. 각 ExecPlan은 `md`로 라벨이 붙은 단 하나의 fenced code block이어야 하며, 삼중 백틱으로 시작하고 끝나야 한다. 그 안에 추가 삼중 백틱 코드 펜스를 중첩하지 않는다. 명령, 출력 기록, diff, 코드를 보여줘야 할 때는 해당 단일 펜스 안에서 들여쓰기 된 블록으로 제시한다. ExecPlan의 코드 펜스가 조기에 닫히는 것을 피하기 위해 ExecPlan 안에서는 코드 펜스 대신 들여쓰기를 사용해 명확히 표현한다. 모든 제목 뒤에는 빈 줄 두 개를 두고, `#`, `##` 등을 사용하며, 순서 있는 목록과 순서 없는 목록에는 올바른 문법을 사용한다.

내용이 단 하나의 ExecPlan뿐인 Markdown(.md) 파일에 ExecPlan을 작성할 때는 삼중 백틱을 생략해야 한다.

쉬운 산문으로 작성한다. 목록보다 문장을 선호한다. 간결함이 의미를 흐리지 않는 한 체크리스트, 표, 긴 열거는 피한다. 체크리스트는 `진행 상황` 섹션에서만 허용되며, 그곳에서는 필수다. 서술 섹션은 산문 중심으로 유지해야 한다.

## 지침

자기완결성과 쉬운 말이 가장 중요하다. 일반 영어가 아닌 표현("daemon", "middleware", "RPC gateway", "filter graph")을 도입한다면 즉시 정의하고, 그것이 이 저장소에서 어떻게 나타나는지 독자에게 다시 알려준다. 예를 들어 해당 표현이 나타나는 파일이나 명령을 이름으로 든다. "앞에서 정의한 대로" 또는 "아키텍처 문서에 따르면"이라고 말하지 않는다. 반복하더라도 필요한 설명을 여기에 포함한다.

흔한 실패 방식을 피한다. 정의되지 않은 전문 용어에 의존하지 않는다. 결과 코드가 컴파일되지만 의미 있는 동작은 하지 않는 방식으로 "기능의 문구"만 지나치게 좁게 설명하지 않는다. 핵심 결정을 독자에게 떠넘기지 않는다. 모호함이 있으면 계획 안에서 해결하고 왜 그 경로를 선택했는지 설명한다. 사용자가 볼 수 있는 효과는 과하다 싶을 만큼 설명하고, 부수적인 구현 세부사항은 덜 지정하는 쪽을 택한다.

관찰 가능한 결과로 계획을 고정한다. 구현 후 사용자가 무엇을 할 수 있는지, 실행할 명령은 무엇인지, 어떤 출력을 봐야 하는지 적는다. 수락 기준은 내부 속성("HealthCheck 구조체를 추가했다")이 아니라 사람이 검증할 수 있는 동작("서버를 시작한 뒤 http://localhost:8080/health 로 이동하면 본문 OK와 함께 HTTP 200을 반환한다")으로 표현해야 한다. 변경이 내부적이라면 그 영향도 어떻게 시연할 수 있는지 설명한다. 예를 들어 변경 전에는 실패하고 변경 후에는 통과하는 테스트를 실행하거나, 새 동작을 사용하는 시나리오를 보여준다.

저장소 맥락을 명시적으로 지정한다. 파일은 저장소 루트 기준 전체 상대 경로로 이름을 쓰고, 함수와 모듈을 정확히 이름으로 부르며, 새 파일을 어디에 만들어야 하는지 설명한다. 여러 영역을 건드린다면 초보자가 자신 있게 탐색할 수 있도록 그 부분들이 어떻게 맞물리는지 설명하는 짧은 방향 안내 문단을 포함한다. 명령을 실행할 때는 작업 디렉터리와 정확한 명령줄을 보여준다. 결과가 환경에 의존한다면 가정을 적고, 합리적인 경우 대안도 제공한다.

멱등적이고 안전하게 작성한다. 단계를 여러 번 실행해도 손상이나 흐트러짐이 생기지 않도록 작성한다. 단계가 중간에 실패할 수 있다면 다시 시도하거나 조정하는 방법을 포함한다. 마이그레이션이나 파괴적 작업이 필요하다면 백업이나 안전한 대안을 명확히 적는다. 진행하면서 검증할 수 있는 추가적이고 테스트 가능한 변경을 선호한다.

검증은 선택 사항이 아니다. 테스트 실행, 필요한 경우 시스템 시작, 유용한 동작 관찰에 대한 지시를 포함한다. 새 기능이나 역량에 대해서는 포괄적인 테스트를 설명한다. 초보자가 성공과 실패를 구분할 수 있도록 예상 출력과 오류 메시지를 포함한다. 가능하면 컴파일을 넘어 변경이 효과적임을 증명하는 방법을 보여준다. 예를 들어 작은 end-to-end 시나리오, CLI 호출, HTTP 요청/응답 기록을 사용한다. 프로젝트 도구 체인에 맞는 정확한 테스트 명령과 그 결과를 해석하는 방법을 적는다.

증거를 포착한다. 단계가 터미널 출력, 짧은 diff, 로그를 만들면 단일 fenced block 안에 들여쓰기 된 예시로 포함한다. 성공을 증명하는 데 집중해 간결하게 유지한다. 패치를 포함해야 한다면, 큰 덩어리를 붙여넣는 대신 독자가 지시를 따라 재현할 수 있는 파일 단위 diff나 작은 발췌를 선호한다.

## 마일스톤

마일스톤은 관료 절차가 아니라 서사다. 작업을 마일스톤으로 나눈다면 각 마일스톤을 짧은 문단으로 소개한다. 범위, 마일스톤이 끝났을 때 이전에는 없던 무엇이 존재하는지, 실행할 명령, 관찰할 것으로 기대하는 수락 기준을 설명한다. 목표, 작업, 결과, 증명이라는 이야기로 읽히게 한다. 진행 상황과 마일스톤은 별개다. 마일스톤은 이야기를 전달하고, 진행 상황은 세부 작업을 추적한다. 둘 다 반드시 존재해야 한다. 단지 짧게 쓰기 위해 마일스톤을 축약하지 말고, 향후 구현에 중요할 수 있는 세부사항을 빠뜨리지 않는다.

각 마일스톤은 독립적으로 검증 가능해야 하며, 실행 계획의 전체 목표를 점진적으로 구현해야 한다.

## 살아 있는 계획과 설계 결정

- ExecPlan은 살아 있는 문서다. 핵심 설계 결정을 내릴 때마다 계획을 업데이트해 결정과 그 배경 생각을 함께 기록한다. 모든 결정은 `결정 기록` 섹션에 기록한다.
- ExecPlan에는 `진행 상황` 섹션, `예상 밖의 발견` 섹션, `결정 기록`, `결과와 회고` 섹션이 있어야 하며 유지해야 한다. 이는 선택 사항이 아니다.
- 접근 방식에 영향을 준 최적화 도구의 동작, 성능 트레이드오프, 예상치 못한 버그, 역방향/되돌리기 의미론을 발견하면 짧은 증거 조각과 함께 `예상 밖의 발견` 섹션에 기록한다. 테스트 출력이 이상적이다.
- 구현 도중 방향을 바꾸면 `결정 기록`에 이유를 문서화하고, 그 영향을 `진행 상황`에 반영한다. 계획은 당신을 위한 체크리스트인 동시에 다음 기여자를 위한 안내서다.
- 주요 작업이나 전체 계획을 완료했을 때는 `결과와 회고` 항목을 작성해 달성한 것, 남은 것, 배운 점을 요약한다.

# 프로토타이핑 마일스톤과 병렬 구현

더 큰 변경의 위험을 낮출 수 있다면 명시적인 프로토타이핑 마일스톤을 포함하는 것은 허용되며, 종종 권장된다. 예를 들어 가능성을 검증하기 위해 의존성에 저수준 연산자를 추가하거나, 최적화 도구 효과를 측정하면서 두 가지 합성 순서를 탐색할 수 있다. 프로토타입은 추가적이고 테스트 가능하게 유지한다. 범위를 "prototyping"이라고 명확히 표시하고, 결과를 실행하고 관찰하는 방법을 설명하며, 프로토타입을 승격하거나 폐기하는 기준을 적는다.

테스트가 계속 통과하도록 추가적 코드 변경을 먼저 하고 그다음 제거하는 방식을 선호한다. 병렬 구현, 예를 들어 마이그레이션 중 이전 경로 옆에 어댑터를 함께 유지하는 방식은 위험을 줄이거나 큰 마이그레이션 중 테스트가 계속 통과하게 할 때 괜찮다. 두 경로를 어떻게 검증할지, 한 경로를 테스트와 함께 안전하게 폐기하는 방법을 설명한다. 여러 새 라이브러리나 기능 영역을 다룰 때는 이러한 기능의 가능성을 서로 _독립적으로_ 평가하는 스파이크를 만드는 것을 고려한다. 외부 라이브러리가 기대대로 동작하고, 우리가 필요한 기능을 독립적으로 구현한다는 점을 증명한다.

## ExecPlan 템플릿

새 ExecPlan을 만들 때는 `EXECPLAN_TEMPLATE.md`를 복사해 시작한다. 템플릿은 이 문서의 요구사항을 실제 계획 파일에 빠짐없이 반영하기 위한 출발점이다. 템플릿의 안내 문구, 예시 문장, 자리 표시자는 그대로 남기지 말고 조사 결과와 구현 맥락에 맞는 구체적인 내용으로 교체해야 한다.

템플릿은 편의를 위한 골격일 뿐이며, 이 문서의 요구사항을 대체하지 않는다. 템플릿을 채운 뒤에도 자기완결성, 관찰 가능한 결과, 검증 방법, 살아 있는 문서 섹션, 의사결정 기록이 충분한지 PLANS.md 전체 기준으로 다시 확인한다.

위 지침을 따르면 단일하고 상태 없는 에이전트 또는 인간 초보자가 ExecPlan을 처음부터 끝까지 읽고 동작하며 관찰 가능한 결과를 만들 수 있다. 기준은 이것이다: 자기완결적이고, 자급자족하며, 초보자를 안내하고, 결과에 집중해야 한다.

계획을 수정할 때는 변경사항이 살아 있는 문서 섹션을 포함한 모든 섹션에 포괄적으로 반영되도록 해야 하며, 계획 맨 아래에는 변경 내용과 변경 이유를 설명하는 메모를 작성해야 한다. ExecPlan은 거의 모든 것에 대해 무엇뿐 아니라 왜도 설명해야 한다.
